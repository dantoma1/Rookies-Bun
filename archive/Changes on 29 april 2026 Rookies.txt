1. RLS enabled on all tables
Previously only company_recruiters had RLS on. Now all 7 tables are protected:
students · jobs · applications · employers · messages · contact_messages · company_recruiters

2. Table policies (29 policies)
students — anyone can read · owner can insert/update/delete · owner cannot change their own is_admin flag

jobs — anyone can read active listings · employers can read/insert/update/delete their own · admins can read all (including inactive)

applications — students can insert and read their own · employers can read and update status on theirs · admins can read all · nobody can delete

messages — students and employers can only read their own threads · each side can only send as themselves · both can mark messages as read

employers — anyone can read · employers can insert/update/delete their own profile · employers cannot change their own status (pending/approved/rejected) · admins can change status

contact_messages — anyone can submit · nobody can read through the app

3. Storage bucket policies
avatars — public read · authenticated owner-only upload/update/delete · 2MB limit · jpg/png/webp only

cvs — authenticated users only can read · authenticated owner-only upload/update/delete · 5MB limit · PDF only · bucket made private

4. Server-side validation (14 constraints)
Rules enforced at database level that can't be bypassed through the API:

Names and titles can't be blank
Email must be a valid format
Message/motivation length caps (2000–5000 chars)
Application status locked to valid values only (New, In review, Shortlisted, Accepted, Rejected)
Job hires between 1–500
Can't apply to the same job twice (unique constraint)
5. Rate limiting (2 triggers)
Applications — max 20 per student per 24 hours
Contact form — max 5 submissions per email per hour
